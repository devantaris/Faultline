🌍 The Big Picture

You know how when WhatsApp is down, suddenly you can't use other apps that rely on it? Or when Amazon's payment page crashes during a sale, the whole checkout breaks? That's called a cascading failure — one thing breaks, and it takes down everything connected to it.

Big companies have hundreds of little programs (called services) running simultaneously — one handles your login, one handles payments, one handles notifications, one handles your order history. They all talk to each other. When one dies, the others panic.

"Imagine a restaurant where the chef, cashier, and delivery guy all depend on each other. If the cashier goes home sick, the chef stops cooking, and the delivery guy sits idle. The whole restaurant breaks — because of one person." That's a cascading failure.

My project, Faultline, lets engineers test this before it happens in real life. You draw your services like boxes on a whiteboard, connect them with arrows, then click a button that says "crash this one." The tool shows you exactly which other boxes break, in what order, and how bad it gets. Then an AI writes a plain-English report telling you what to fix.

Without Faultline

With Faultline

Engineers test failures safely beforehand. Fix weak points. Build confidence before launch.

📐 The Formal Problem

Faultline addresses the fault propagation problem in directed acyclic service dependency graphs. Given a graph G = (V, E) where vertices represent microservices and directed edges represent synchronous call dependencies, the system models how a localized fault at node v₀ propagates through the graph based on configurable per-node resilience parameters.

The propagation algorithm performs a modified BFS traversal in reverse-dependency order — evaluating for each caller whether its effective latency budget is exceeded by the sum of base latency, failure-induced delay, and retry amplification. Nodes that exceed their timeout trigger state transitions in dependent nodes, creating a deterministic failure chain.

📚 Concepts Implemented

  • Circuit Breaker Pattern — 3-state FSM (CLOSED/OPEN/HALF-OPEN) with configurable failure threshold, recovery timeout, and probe logic. Based on Michael Nygard's Release It! and Netflix's Hystrix implementation.
  • Retry Storm Modelling — Amplification factor computed as retry_count × concurrent_callers. Models the non-obvious failure mode where latency-induced retries cause overload faultlines.
  • Timeout Budget Inheritance — Each service has a timeout budget. Effective response time = baseline + failure penalty. If budget exceeded, caller fails — propagating upward.
  • Stochastic Packet Loss — Probabilistic Bernoulli trial per request at configurable drop rate. Retries interact with timeout budget to determine failure threshold.
  • LLM Structured Output — Gemini function-calling (not free text) maps simulation telemetry to typed incident report schema. Avoids hallucination by grounding in real simulation data.

🧪 What This Is (and Isn't)

Faultline is a deterministic simulation model for failure propagation in microservice dependency graphs. It is not a live chaos engineering tool (that would require actual service injection, like Gremlin or Chaos Monkey). It is a configurable model that lets engineers reason about failure blast radius before deployment.

The simulation engine uses a graph-traversal approach with parameterized resilience primitives per node. The failure physics are approximations — the retry storm amplification model assumes uniform concurrency, which is unrealistic in production. The circuit breaker FSM uses a simplified rolling-window failure counter rather than a leaky-bucket or token-bucket rate limiter.

🔍 Honest Limitations

  • Assumes static topology — no dynamic service discovery or autoscaling modelled
  • Retry storm amplification is a linear model; real production exhibits non-linear feedback loops
  • No partial failure states — nodes are binary (alive/dead) with a degraded intermediate
  • LLM report generation is bounded by simulation output quality; garbage in, garbage out
  • No temporal correlation of failures — multi-site, simultaneous failure injection not supported

🚀 Open Research Directions

  • Probabilistic fault tree analysis integration for quantitative reliability scoring
  • Monte Carlo simulation across failure scenarios to compute expected blast radius distributions
  • Comparison against production incident data to validate simulation accuracy
  • Integration with OpenTelemetry traces to learn real-world latency distributions per edge

💬 The Honest Technical Breakdown

So basically — the frontend is React + React Flow for the graph canvas. React Flow is actually really good, you can make custom nodes and it handles drag-and-drop natively. Each node has a side panel for config (timeout, retry count, circuit breaker on/off). The graph gets serialized to JSON and sent to a FastAPI backend.

The backend is where it gets interesting. I'm using NetworkX to model the service graph as a DiGraph. When you inject a failure, it runs a BFS from that node going upward through callers. For each caller it computes: does the effective latency (base + failure penalty × retries) exceed the caller's timeout? If yes, that caller fails too. Circuit breaker FSM intercepts if it's enabled — CLOSED → OPEN → HALF-OPEN. It returns a timeline of what died when.

Then that structured output goes to Gemini — but with tool calling, not a plain prompt. I defined a function schema, Gemini fills it in. The frontend renders the filled schema as the blast radius report.

😅 What Was Actually Hard

  • The retry storm logic — getting the amplification math right took way longer than expected
  • React Flow re-renders — you have to memo aggressively or the canvas lags on big graphs
  • Gemini tool calling — the schema definition syntax is finicky, errors are cryptic
  • Animating the faultline in order on the graph — had to build a custom BFS-timed animation queue
  • Making packet loss feel realistic — pure probability felt too random, had to add burst correlation

🧠 What I'd Tell You to Learn First

  • Read about circuit breakers before touching the code — the pattern makes way more sense conceptually first
  • NetworkX is incredibly simple for graph stuff, 10 minutes to get productive
  • React Flow has great docs, start with their custom node example
  • Gemini tool calling: define your schema first, test it in AI Studio before wiring it to your backend
