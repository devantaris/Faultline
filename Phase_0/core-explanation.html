What Is Faultline?

The core idea in one sentence

Faultline is a browser-based tool that lets you visually construct a microservices dependency graph, inject specific failure types into any service, and watch a physics-accurate simulation of how that failure propagates — producing an AI-generated blast radius report at the end.

Why it matters: In distributed systems, failures are rarely local. A 200ms latency spike in a payment API causes retries, which cause a retry storm, which exhausts connection pools, which crashes the orders service, which triggers alerts, which wakes up 3 engineers at 3am. Faultline lets you discover this chain before production does.

System 1

Topology Builder

The canvas where users construct their microservices architecture. Built on React Flow, it provides a drag-and-drop interface with custom node types and configurable edge properties.

Node Properties

  • timeout_ms — request timeout threshold
  • retry_count — max retries before fail
  • circuit_breaker — on/off toggle
  • cb_threshold — failure % to trip breaker
  • instance_count — replicas running
  • memory_limit_mb — resource cap

Edge Properties

  • latency_ms — baseline network delay
  • protocol — HTTP, gRPC, async
  • dependency_type — hard vs soft
  • traffic_weight — % of calls going this path

Hard dependencies: if target fails, caller fails. Soft: caller degrades but survives.

System 2

Failure Injection Engine

Four failure modes, each with distinct propagation physics. The user selects a node, picks a failure type, configures severity, and triggers the simulation.

💥

Service Crash

Hard node removal. All callers immediately see connection refused. Circuit breakers at callers trip if configured. Propagation: instant, breadth-first.

🐢

Network Delay

Adds configurable latency (ms) to all responses. Callers accumulate wait time. Timeout chain triggers if total > caller's timeout_ms. Causes retry amplification.

📦

Packet Loss

Probabilistic drop rate (e.g. 40%). Each request has a dice roll. Failed packets trigger retries. If retry_count * latency > timeout → failure. Faultlines stochastically.

🧠

Memory Spike

Consumes memory up to limit. GC pressure causes latency jitter. If OOM: crash. Affects concurrency capacity. Slowest propagation — gradual degradation pattern.

Key insight: Different failure types produce radically different blast shapes. A crash is a sharp spike. A memory leak is a slow curve. A delay causes a retry storm (the most dangerous and least intuitive). Faultline shows all of these visually.
System 3

Faultline Simulation Engine

The backend Python engine that takes the graph + failure seed and runs the propagation simulation. This is the intellectual core of the project.

Graph Representation

Services = nodes. Calls = directed edges. NetworkX DiGraph internally. Topological sort determines evaluation order. BFS from failure seed node.

Propagation Algorithm

def simulate_faultline(graph, failed_node, failure_type, severity):
    queue = deque([failed_node])
    visited = {failed_node: "DEAD"}
    timeline = []
    t = 0

    while queue:
        node = queue.popleft()
        callers = graph.predecessors(node)  # who calls this node?

        for caller in callers:
            edge = graph[caller][node]
            # calculate effective latency seen by caller
            effective_latency = compute_effective_latency(
                node, failure_type, severity, edge["latency_ms"]
            )
            # check retry storm amplification
            total_time = effective_latency * caller["retry_count"]

            if total_time > caller["timeout_ms"]:
                if caller["circuit_breaker"]:
                    status = evaluate_circuit_breaker(caller)
                else:
                    status = "DEAD"
                t += total_time / 1000  # convert to seconds
                timeline.append({"node": caller, "status": status, "t": t})
                if status == "DEAD":
                    queue.append(caller)

    return timeline

Circuit Breaker State Machine

CLOSED ✓

Normal operation. Calls flow through. Failure rate tracked in rolling window.

OPEN ✗

Threshold exceeded. All calls fail-fast. No requests to downstream. Timer starts.

HALF-OPEN ◑

Trial probe sent. If success → CLOSED. If fail → OPEN again. Recovery path.

Retry Storm Logic

This is the most dangerous and least understood failure mode. When a service is slow (not dead), callers retry. Each retry adds load. More load → more slowness → more retries. Exponential explosion.

# Retry storm amplification factor
amplification = retry_count ** concurrent_callers
# e.g. 3 retries × 10 caller instances = 30x load spike
# If original service handled 100 rps, now it sees 3000 rps
# This is why delays are more dangerous than crashes
System 4

Blast Radius AI Report

After simulation, the structured output (timeline, affected nodes, user impact %) is sent to Gemini with a system prompt + tool definitions. Gemini generates a plain-English incident report.

Gemini Tool Schema

generate_blast_report({
  affected_services: string[],
  faultline_timeline_s: number[],
  user_impact_pct: number,
  recovery_estimate_s: number,
  root_cause: string,
  contributing_factors: string[]
})

Sample Output

"Your Payment API failure faultlines to the Orders Service in 4.2s via retry storm. The Auth Service circuit breaker trips at 8s, protecting 32% of users. Full outage affects 68% of sessions. Estimated manual recovery: 8–14 minutes."

Why tool-calling matters: Using Gemini's function calling (not free-text) guarantees structured, typed output. The report is generated from real simulation data — not hallucinated. This is the correct way to use LLMs in engineering systems.
